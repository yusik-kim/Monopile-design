# Monopile Foundation Concept Design Engine — Methodology Log

## Decisions
- 2026-07-16 — Adopted Arany et al. (2017) "10-step" ULS→SLS→NFA→FLS iterative sizing loop — matches published industry concept-stage practice.
- 2026-07-16 — Soil-structure interaction: closed-form Hetenyi beam-on-elastic-foundation (idealized homogeneous sand/clay), not full p-y/PISA — user confirmed speed over accuracy for concept stage.
- 2026-07-16 — ULS: single blended load factor γf=1.35 (DNV normal safety class) instead of separate wind/wave partial factors — no detailed load-case combinations at concept stage.
- 2026-07-16 — FLS: single DNV-RP-C203-style S-N curve (log10(a)=12.16, m=3) with DFF=2.0, not joint-specific curves — no detailed joint design exists yet.
- 2026-07-16 — SLS governed by mudline rotation only (0.5° default), no separate deflection limit — matches the industry-recognized governing SLS criterion.
- 2026-07-16 — Natural frequency: 2-spring (K_L, K_R) model, omitting K_LM cross-coupling — kept the derivation self-verifiable in closed form rather than risk misquoting Arany's published 3-spring coefficients.
- 2026-07-16 — Tower stiffness regression calibrated to the published IEA 15MW tower geometry (base 10m/top 6.5m), sole anchor point. Backfill detail: the earlier uncalibrated version (`4.0 + 0.01×hub_height`) underestimated tower stiffness so severely that tower flexibility dominated >95% of total system flexibility at all realistic pile sizes — the sizing loop grew pile diameter to 84m chasing a frequency target pile geometry couldn't move, since the model was blaming the foundation for a problem that was actually in the tower term.
- 2026-07-16 — FATIGUE_LOAD_FACTOR changed 0.35→0.17 (ad-hoc) — 0.35 made FLS demand ~2x too much wall thickness vs. the published IEA 15MW reference (55mm); 0.17 back-solved to match it. User: recalibrate properly once the model is more mature.
- 2026-07-16 — Replaced 6 hand-estimated TURBINE_LIBRARY entries (8-20MW) with 5 entries sourced from real reference-turbine reports (5/10/15/22 MW verified + 25MW extrapolated) — see METHODOLOGY_REPORT.md §2 for sources (OC3/NREL 5MW, DTU 10MW, IEA 15MW, IEA 22MW).
- 2026-07-16 — dt_ratio_max widened 140→160 — matches the real D/t optimization bound used in the IEA 22MW reference monopile design.
- 2026-07-16 — RNA/tower mass split (natural-frequency check only) changed 40/60→50/50 — real observed range across 4 sourced turbines is 43.6%-54.2% RNA fraction; 50/50 is the average, not a fixed ratio.
- 2026-07-16 — NFA cantilever changed from one uniform "average tower" section over the whole mudline-to-hub span to a two-segment model (real pile EI below the transition piece, average-tower EI above it) — the uniform version lumped the much-stiffer pile-above-mudline length in with the flexible tower EI, systematically underpredicting f0 (found via the 5MW/OC3 real-geometry check). Added `transition_piece_height_m` to TURBINE_LIBRARY (sourced: OC3=10m, IEA15/22MW=15m each; DTU10MW assumed=15m, not stated in its turbine-only report).
- 2026-07-16 — Soft-stiff band formula: added a fallback for when `3×rpm_min < 1.1×rpm_max` (band would invert) — falls back to a lower-bound-only criterion using `3×rpm_max` as a loose ceiling, matching how the IEA 22MW report itself frames its frequency target (single-sided 0.15Hz minimum, no explicit 3P-avoidance). Fixes the 22MW band inversion; result now matches the report's stated ~0.16Hz achieved frequency to 3 s.f. Backfill detail: before these two fixes (this entry + the two-segment cantilever above), evaluating the model at the *real* 5MW/OC3 geometry (D=6m, t=60mm, L=36m) gave f0=0.191Hz, below its own target band [0.222, 0.31]Hz; at the *real* 22MW/IEA geometry (D=10m, t=100mm, L=45m) the target band itself came out inverted (band_low=0.1295 > band_high=0.0813). Both were caught because `size_monopile` failed to converge and flagged it, rather than silently returning a wrong answer.
- 2026-07-16 — FATIGUE_LOAD_FACTOR changed 0.17→0.24, after a sensitivity sweep (0.17→0.35 step 0.02, refined 0.005 near the transition) at the 15MW/35m/sand reference case. At ≤0.23, FLS utilization scales as `factor^3` (matches the S-N slope m=3) with geometry unchanged (D=9.33m, t=84.8mm, NFA-governed); FLS crosses over to become the governing check at 0.24 (utilization 0.911) without requiring any wall-thickness increase. Chosen because industry experience (and this project's own research summary) expects FLS to typically govern monopile sizing, which 0.17 contradicted (FLS=0.324, well below ULS/NFA). Still ad-hoc — not derived from a DLC spectrum — but no longer produces a governing-check pattern that contradicts known practice at the reference case.
- 2026-07-16 — User flagged doubt about the NFA check's validity (separate from whether its frequency formula matches published values — see METHODOLOGY_REPORT.md §7/§10) and asked to redo the FATIGUE_LOAD_FACTOR calibration *excluding* NFA, targeting the true minimum wall thickness (not the grow-only loop's output) for the 15MW case. Sensitivity sweep at fixed D=9.33m/L=46.67m (bisecting for the smallest thickness satisfying ULS+SLS+FLS only, NFA excluded from gating) found: t=50mm needs FLF≈0.1475 but implies D/t=186.7, exceeding the existing dt_ratio_max=160 bound; t=60mm needs FLF≈0.1765, D/t=155.5, within bounds, close to the real IEA 15MW reference thickness (~55mm). FATIGUE_LOAD_FACTOR set to **0.176**.
- 2026-07-16 — Discovered while applying the above: `size_monopile`'s growth loop is one-directional (only adds material, never removes it) and starts from `t0=D/110`. If that guess already passes every check, the loop runs zero iterations, so the 0.176 value alone didn't change the 15MW case's output (still 84.8mm, NFA "governing" only because it had the least margin, not because it was forcing anything). Fixed by adding a post-convergence thickness-shrink step to `size_monopile` (steps wall thickness down by t_step while all four checks stay ≤1.0, until a further step would fail one or the dt_ratio_max floor is hit). Diameter is deliberately not shrunk, pending NFA's own verification. This changed the converged thickness for **every** verification case (not just 15MW) — see METHODOLOGY_REPORT.md §9/§10. Consequence: NFA margins are now much tighter across the board (as low as 0.002 for the 8MW/clay case), raising the priority of NFA's pending verification.
- 2026-07-17 — Investigated buckling (see METHODOLOGY_REPORT.md §5/§11 item 18) and decided **not** to implement it in `engine.py` at this stage. Global (Euler) buckling: very unlikely to govern — axial load is tiny relative to a large-diameter shell's elastic critical load. Local shell buckling: checked by hand (DNV-RP-C202, cross-checked against WISDEM's `utilization_dnvgl.CylinderBuckling`) at the 15MW reference case — a single unstiffened panel over the full 35m exposed water column gives utilization≈2.2, but that's unrealistic (WISDEM itself checks per discretized section, not the whole shaft); at a more realistic 10m section length, utilization≈1.07 — close but not decisively governing. Decision: since concept stage uses one constant wall thickness while a real design would taper it (thicker at mudline, thinner toward the interface), refining a buckling check now would be precision spent on a thickness distribution later stages will change anyway.
- 2026-07-17 — Sensitivity test for 5MW/22MW (buckling included, panel length 2-30m) showed buckling only closes part of the gap to real reference thicknesses at realistic panel lengths; matching the real values needs `l≈55-60m`, which exceeds the actual physical exposed shaft length for both turbines (30m for 5MW, 49m for 22MW). Conclusion: missing buckling is a real contributor but **not the sole explanation** for why this tool's D/t lands at 160 instead of the real ~100 — other unmodeled factors (DNVGL's separate gamma_f/gamma_m/gamma_n/gamma_b partial factors vs. this tool's single blended GAMMA_F_ULS, missing Morison inertia term, no corrosion allowance, no driveability floor, rough self-weight estimate) likely also contribute.
- 2026-07-17 — Fixed a stale `_initial_geometry` bug: the diameter initial-guess formula was still anchored to `7.0m at 8MW / 11.0m at 20MW`, leftover from the *original* six-turbine hand-estimated library that `TURBINE_LIBRARY` replaced on 2026-07-16 with sourced 5/10/15/22/25 MW anchors — never updated to match, so it extrapolated off turbine data that no longer existed in the code. Found while investigating a Streamlit run showing `L/D` exactly 5.000 and `D/t` exactly 160 for every turbine size: the growth loop runs zero iterations whenever the initial guess already satisfies every check (true for almost every turbine size tested), so those weren't converged/solved values, just the untouched starting guess. Fixed by re-anchoring to the two best-sourced real reference diameters: 6.0m at 5MW, 10.0m at 15MW. Result: 15MW's diameter now lands at exactly the real IEA reference (10.0m), but this also means FLS no longer governs there (bending stress drops with the larger, now-correct diameter) — every real-turbine verification case now sits at D/t=160 with NFA the only check with any margin left.
- 2026-07-17 — Removed the 20MW verification row (interpolated turbine, no real reference design to compare against) and the 8MW/clay synthetic test case (`check_8mw_clay_converges_since_nfa_fix` in test_engine.py — built to regression-test the beta*L<2.5 flag and a past NFA convergence bug, not to reproduce a real design) — user's explicit choice, to keep verification limited to cases with genuine reference-design comparisons (5/15/22 MW).
- 2026-07-18 — **Implemented local shell buckling** (`_shell_buckling_check`, DNV-RP-C202 unstiffened cylinder) as a genuine 5th check in `evaluate_monopile`/`size_monopile`, reversing the 2026-07-17 "decided not to implement" call. `l_panel` = the exposed above-mudline shaft length (water depth + freeboard), not can length -- since these monopiles are assumed to have no dedicated ring stiffeners, per the earlier sensitivity-test finding. Result: converged thickness rose substantially for all three verified cases (5MW 37.5mm->54.5mm, 15MW 62.5mm->100.9mm, 22MW 80.0mm->110.4mm); buckling now governs or nearly governs all three, resolving the D/t=160-ceiling problem the anchor fix had exposed. Also added a comment in `_shell_buckling_check` noting the pre-buckling 15MW result (t~=50mm) for historical comparison.
- 2026-07-18 — **Recalibrated FATIGUE_LOAD_FACTOR 0.176 -> 0.3.** With buckling now governing 5/15/22MW's converged geometries, solved for the FLF that makes FLS=1.0 exactly at each: 0.282 (5MW), 0.336 (15MW), 0.547 (22MW), average 0.389. User chose a rounder **0.3** instead, as a deliberately conservative concept-design number -- higher than the pre-buckling 0.176, but not uniformly conservative across all three cases (below both the 15MW and 22MW individual fits, and below the three-case average).
- 2026-07-18 — **Recalibrated again, FATIGUE_LOAD_FACTOR 0.3 -> 0.35**, based on a full sensitivity sweep (0.25-0.60, step 0.05) with buckling active throughout for 5/15/22MW. Finding: the FLS-vs-Buckling crossover point scales with turbine size (~0.30 for 5MW, ~0.35 for 15MW, ~0.55 for 22MW) -- there is no single FLF that makes FLS govern uniformly across all three. User chose **0.35** empirically because: (a) FLS governs both 5MW and 15MW there (t=64.5mm, 106.9mm), matching industry expectation that fatigue drives sizing below ~15MW; (b) 22MW stays buckling-governed (t=110.4mm), treated as expected large-turbine behavior rather than a contradiction; (c) resulting thicknesses land within ~10% of the real OC3 5MW (60mm) and IEA 22MW (100mm) references -- notably closer than the 0.3 calibration.

- 2026-07-18 — Investigated the "zero iterations" pattern further: found it also applies to diameter (and therefore embedded length, via the fixed `L/D=5` ratio at the initial guess). The growth loop only grows `D` when NFA/ULS/FLS/SLS is actively failing at the current geometry; for most turbine sizes, `_initial_geometry`'s starting guess already satisfies every check, so `D` never moves either. Before local shell buckling was implemented, this meant most verification cases converged to exactly `D/t=160` (the `dt_ratio_max` ceiling from the post-convergence thickness-shrink step) with `L/D` exactly 5.000 (the untouched initial-guess ratio) — a result that could look like a solved, optimized design while actually being almost entirely the raw starting guess plus one artificial ceiling. Implementing buckling (previous entry) resolved most of this for the three verified cases, since buckling now actively binds before the `dt_ratio_max` ceiling is reached.
- 2026-07-19 — **15 MW Fig. 4-2 mudline thickness estimate (~50mm, image-derived from the IEA 15MW report's diameter/thickness-vs-height chart) is now doubted.** This engine's own buckling-inclusive converged 15MW design lands at D/t≈93-99, matching the 5MW/22MW *sourced* references' D/t≈93-116 pattern closely, while the Fig. 4-2 estimate implies D/t≈200 (much thinner) — an outlier relative to the other two real designs. Most likely explanation: a misread of the chart (e.g. picking a point above the mudline, where the real design's wall is genuinely thinner, rather than at the mudline itself) rather than a real ~50mm mudline value. Not corrected in the turbine-library cross-check table pending an actual table-sourced number for 15MW; flagged as doubtful instead.

- 2026-07-20 — **D/t changed from a hard search clamp to an advisory bound with a warning note.** Previously `_clamped()` forcibly reset wall thickness every iteration to keep `D/t` inside `[dt_ratio_min, dt_ratio_max]`, which is exactly what caused the "converges to exactly D/t=160" artifact noted above (2026-07-18 entry) — the clamp was silently overriding what the physics checks actually needed. Now the growth/shrink loop lets `D/t` move freely, and `size_monopile` appends a note (`"D/t = {n} is beyond general manufacturability requirements ({min} < D/t < {max})"`) if the final geometry ends up outside the configured range, rather than blocking the search from getting there. L/D bounds are still a hard clamp (unchanged) — only D/t changed. User's explicit request: let the tool continue rather than stop at the boundary.

- 2026-07-20 — **Water depth input widget max raised 80m → 150m; `l_over_d_max` default raised 8 → 12.** User reported the Streamlit app "stopped updating" past 80m water depth — root cause was `app.py`'s `number_input` widget itself capping at `max_value=80.0` (a UI-only limit, not an engine constraint), not the L/D bound as initially suspected. Fixed the widget cap, and separately widened `l_over_d_max` (both the `DesignInputs` default and the app's "Max L/D" widget default) from 8 to 12, since deep water genuinely needs longer embedment relative to diameter and the old default was needlessly tight for the newly-reachable depth range. Verified: 15MW/150m/sand converges (D=11.65m, t=292.9mm, L/D=4.29, Buckling-governed, D/t=40 correctly flagged as beyond manufacturability guidance) with no regression on the three real-turbine benchmark cases.
- 2026-07-20 — **User observed embedded length `L` doesn't grow with water depth/moment, even at the new 150m range, and asked why.** Root cause has two layers: (1) mechanically, `size_monopile`'s iteration loop never picks `L` as a lever -- it starts at a fixed `L0=5*D0` ratio and only moves as a passive side effect of the `L/D` clamp reacting to `D`; (2) more fundamentally, the closed-form `K_L`/`K_R` pile-head stiffness terms driving SLS/NFA have **zero dependence on `L` for clay** (constant `k_soil`) and only a weak one for sand (via `z_ref=L/3`) -- a genuine property of the semi-infinite-beam solution once `beta*L` clears its validity threshold, not a coding gap. So even a wired-up "SLS failing -> grow L" branch would not fix this for clay. **Decision: document as future work rather than implement now** -- the real fix is an ultimate lateral/moment soil *capacity* check (e.g. Broms' method, or integrated ultimate p-y resistance), which is a different kind of check from anything currently implemented and a larger addition than a quick wire-up. Added explicit "FUTURE WORK" comments in `engine.py` (`_soil_stiffness` and `size_monopile` docstrings) and expanded `docs/METHODOLOGY_REPORT.md` Section 9 and assumption item 21 with the clay-insensitivity finding, so this doesn't need rediscovering later.
- 2026-07-23 — **Added a diameter-shrink step to `size_monopile`, closing the gap flagged since 2026-07-19: the growth loop only ever grows `D` and stops at the first passing geometry, so `size_monopile` was never actually verified to return a minimum diameter.** Discovered concretely while comparing BC90 (on the `bc90-extension` branch) against this baseline: holding `t`/`L` fixed at a 75m-water-depth/15MW baseline's own values and just shrinking `D` from 10.00m, every one of the five checks stayed <=1.0 down to D=9.15m before FLS crossed 1.0 -- diameter headroom available with zero mooring involved. Fixed by adding a diameter-shrink loop (same greedy pattern as the existing thickness-shrink step), positioned before it. Reverted the same day (see next entry) after a turbine-capacity sweep found it was implicated (alongside a separate, independent bug) in physically implausible results.
- 2026-07-23 — **Found and fixed the real bug behind "D changes non-incrementally as turbine capacity increases"; the diameter-shrink step above was reverted then re-added in the process, and the 22MW verification case's numbers changed as a result.** A 5-25MW turbine-capacity sweep (user-requested) showed `size_monopile` collapsing to physically implausible geometries (D/t as low as ~4, one case not converging) around 17-19MW. Root cause, traced step-by-step: the growth loop's NFA-too-stiff branch (`elif checks["NFA"] > 1.0: shrink D`, present since before the diameter-shrink step above and independent of it) shrank `D` unconditionally, fighting the ULS/FLS/Buckling branch's "grow t" response -- the two spiraled for ~100 iterations into an unrealistic D/t ratio. Confirmed the diameter-shrink step itself was NOT the cause by reverting it (`git revert`) and reproducing the identical pathology on the reverted code. Fix: the NFA-too-stiff branch now checks whether shrinking `D` further would itself push ULS/FLS/Buckling above 1.0, and stops if so; if the stop is specifically the too-stiff case (not just any NFA failure), the geometry is accepted as converged with a warning note instead of flagged non-convergent, since stiff-side `f0` exceedance (above 3P minimum) is not normally safety-critical for monopiles. Applied to master and `bc90-extension` identically. The diameter-shrink step was then re-added to master (2026-07-24) now that the actual bug was fixed independently. Net effect on the verification table (METHODOLOGY_REPORT.md §10): 5MW and 15MW/35m unchanged, but **22MW now converges to D=10.10m/t=114.4mm/FLS-governed (was D=12.80m/t=110.4mm/Buckling-governed)** -- closer to the real IEA 22MW reference (10.0m) than before, because the now-fixed growth loop reaches convergence at a different point, giving the post-convergence diameter-shrink pass a different (more favorable) starting geometry to work from.
- 2026-07-24 — **Compared the fixed post-convergence order (diameter-then-thickness) against reversing it (thickness-then-diameter) and a joint/coordinated search, user-requested, to check whether the greedy single-order shrink finds the true minimum.** Not a code change. Result: for most cases tested (5MW/20m, 15MW/35m, 15MW/90m), all three approaches land on the identical geometry, because whichever check is tightest at every point along the shrink path also happens to be one that thinning `t` makes worse (verified numerically: e.g. 15MW/90m has Buckling=0.997 at the start, thinning `t` by one step immediately pushes it to 1.023) -- so `t` never has a legal move regardless of order, and only `D` ever actually shrinks. But one counter-example was found (22MW/34m): reversing the order finds a geometry ~2% lighter than the current D-then-t order. Confirms the fixed order is a real, if usually inconsequential, simplification -- it finds *a* smaller passing geometry, not provably *the* smallest -- documented in METHODOLOGY_REPORT.md §9/§11 item 20 rather than changed, since the counter-example is minor and no scratch script for this was kept as a permanent tool.

## Open questions
- **Embedded length `L` has no capacity-driven growth mechanism** (found 2026-07-20, documented as future work, not yet implemented) -- implementing an ultimate lateral/moment soil capacity check (Broms' method is the standard concept-stage choice) and wiring it into `size_monopile` as a new lever for `L` would let embedment respond to load/water depth the way real designs do. Until then, deeper water or larger moments only grow diameter/thickness in this model, never embedment.
- FATIGUE_LOAD_FACTOR=0.35 is still ad-hoc (chosen empirically from a sensitivity sweep to balance "FLS governs below ~15MW" against reference-thickness matching -- see above -- not derived from a DLC spectrum) — recalibrate later against real fatigue load data.
- **NFA is not yet verified** (user-flagged, 2026-07-16) — separate from the f0-formula check already done (§7/§10 below). With buckling now implemented, NFA has more margin than before in all three verified cases (buckling/FLS carry more of the governing load), which reduces but doesn't eliminate how much a future NFA correction could move these results. Now that `size_monopile` also shrinks diameter (2026-07-23 entry above), NFA governing a diameter-shrunk case is flagged in `result.notes` -- but the underlying verification gap itself is still open.
- Global (Euler) column buckling remains unimplemented -- investigated 2026-07-17, found very unlikely to govern; lower priority than local shell buckling was.
- Sand nh-vs-friction-angle table and clay k=0.25×su correlation are rough approximations, not validated against site-specific geotechnical data.
- NFA's f0 prediction is checked against real reference geometries at 5/22MW (sourced) and informationally at 15MW (estimated thickness) — see METHODOLOGY_REPORT.md §10 — but still omits the K_LM cross-coupling term and uses a single-point-calibrated tower regression for the tower segment above the transition piece; the check as a whole is not yet considered verified (see above).
- `_initial_geometry`'s diameter formula is only anchored at 5MW/15MW; for turbine sizes away from those two points (e.g. 22MW, 25MW) it's an unclamped linear extrapolation, not independently validated.
