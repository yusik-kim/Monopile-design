# Monopile Foundation Concept Design Engine — Methodology Log

## Decisions
- 2026-07-16 — Adopted Arany et al. (2017) "10-step" ULS→SLS→NFA→FLS iterative sizing loop — matches published industry concept-stage practice.
- 2026-07-16 — Soil-structure interaction: closed-form Hetenyi beam-on-elastic-foundation (idealized homogeneous sand/clay), not full p-y/PISA — user confirmed speed over accuracy for concept stage.
- 2026-07-16 — ULS: single blended load factor γf=1.35 (DNV normal safety class) instead of separate wind/wave partial factors — no detailed load-case combinations at concept stage.
- 2026-07-16 — FLS: single DNV-RP-C203-style S-N curve (log10(a)=12.16, m=3) with DFF=2.0, not joint-specific curves — no detailed joint design exists yet.
- 2026-07-16 — SLS governed by mudline rotation only (0.5° default), no separate deflection limit — matches the industry-recognized governing SLS criterion.
- 2026-07-16 — Natural frequency: 2-spring (K_L, K_R) model, omitting K_LM cross-coupling — kept the derivation self-verifiable in closed form rather than risk misquoting Arany's published 3-spring coefficients.
- 2026-07-16 — Tower stiffness regression calibrated to the published IEA 15MW tower geometry (base 10m/top 6.5m), sole anchor point — an earlier uncalibrated version caused a sizing-loop divergence bug (see METHODOLOGY_REPORT.md §10).
- 2026-07-16 — FATIGUE_LOAD_FACTOR changed 0.35→0.17 (ad-hoc) — 0.35 made FLS demand ~2x too much wall thickness vs. the published IEA 15MW reference (55mm); 0.17 back-solved to match it. User: recalibrate properly once the model is more mature.
- 2026-07-16 — Replaced 6 hand-estimated TURBINE_LIBRARY entries (8-20MW) with 5 entries sourced from real reference-turbine reports (5/10/15/22 MW verified + 25MW extrapolated) — see METHODOLOGY_REPORT.md §2 for sources (OC3/NREL 5MW, DTU 10MW, IEA 15MW, IEA 22MW).
- 2026-07-16 — dt_ratio_max widened 140→160 — matches the real D/t optimization bound used in the IEA 22MW reference monopile design.
- 2026-07-16 — RNA/tower mass split (natural-frequency check only) changed 40/60→50/50 — real observed range across 4 sourced turbines is 43.6%-54.2% RNA fraction; 50/50 is the average, not a fixed ratio.
- 2026-07-16 — NFA cantilever changed from one uniform "average tower" section over the whole mudline-to-hub span to a two-segment model (real pile EI below the transition piece, average-tower EI above it) — the uniform version lumped the much-stiffer pile-above-mudline length in with the flexible tower EI, systematically underpredicting f0 (found via the 5MW/OC3 real-geometry check). Added `transition_piece_height_m` to TURBINE_LIBRARY (sourced: OC3=10m, IEA15/22MW=15m each; DTU10MW assumed=15m, not stated in its turbine-only report).
- 2026-07-16 — Soft-stiff band formula: added a fallback for when `3×rpm_min < 1.1×rpm_max` (band would invert) — falls back to a lower-bound-only criterion using `3×rpm_max` as a loose ceiling, matching how the IEA 22MW report itself frames its frequency target (single-sided 0.15Hz minimum, no explicit 3P-avoidance). Fixes the 22MW band inversion; result now matches the report's stated ~0.16Hz achieved frequency to 3 s.f.
- 2026-07-16 — FATIGUE_LOAD_FACTOR changed 0.17→0.24, after a sensitivity sweep (0.17→0.35 step 0.02, refined 0.005 near the transition) at the 15MW/35m/sand reference case. At ≤0.23, FLS utilization scales as `factor^3` (matches the S-N slope m=3) with geometry unchanged (D=9.33m, t=84.8mm, NFA-governed); FLS crosses over to become the governing check at 0.24 (utilization 0.911) without requiring any wall-thickness increase. Chosen because industry experience (and this project's own research summary) expects FLS to typically govern monopile sizing, which 0.17 contradicted (FLS=0.324, well below ULS/NFA). Still ad-hoc — not derived from a DLC spectrum — but no longer produces a governing-check pattern that contradicts known practice at the reference case.
- 2026-07-16 — User flagged doubt about the NFA check's validity (separate from whether its frequency formula matches published values — see METHODOLOGY_REPORT.md §7/§10) and asked to redo the FATIGUE_LOAD_FACTOR calibration *excluding* NFA, targeting the true minimum wall thickness (not the grow-only loop's output) for the 15MW case. Sensitivity sweep at fixed D=9.33m/L=46.67m (bisecting for the smallest thickness satisfying ULS+SLS+FLS only, NFA excluded from gating) found: t=50mm needs FLF≈0.1475 but implies D/t=186.7, exceeding the existing dt_ratio_max=160 bound; t=60mm needs FLF≈0.1765, D/t=155.5, within bounds, close to the real IEA 15MW reference thickness (~55mm). FATIGUE_LOAD_FACTOR set to **0.176**.
- 2026-07-16 — Discovered while applying the above: `size_monopile`'s growth loop is one-directional (only adds material, never removes it) and starts from `t0=D/110`. If that guess already passes every check, the loop runs zero iterations, so the 0.176 value alone didn't change the 15MW case's output (still 84.8mm, NFA "governing" only because it had the least margin, not because it was forcing anything). Fixed by adding a post-convergence thickness-shrink step to `size_monopile` (steps wall thickness down by t_step while all four checks stay ≤1.0, until a further step would fail one or the dt_ratio_max floor is hit). Diameter is deliberately not shrunk, pending NFA's own verification. This changed the converged thickness for **every** verification case (not just 15MW) — see METHODOLOGY_REPORT.md §9/§10. Consequence: NFA margins are now much tighter across the board (as low as 0.002 for the 8MW/clay case), raising the priority of NFA's pending verification.
- 2026-07-17 — Investigated buckling (see METHODOLOGY_REPORT.md §5/§11 item 18) and decided **not** to implement it in `engine.py` at this stage. Global (Euler) buckling: very unlikely to govern — axial load is tiny relative to a large-diameter shell's elastic critical load. Local shell buckling: checked by hand (DNV-RP-C202, cross-checked against WISDEM's `utilization_dnvgl.CylinderBuckling`) at the 15MW reference case — a single unstiffened panel over the full 35m exposed water column gives utilization≈2.2, but that's unrealistic (WISDEM itself checks per discretized section, not the whole shaft); at a more realistic 10m section length, utilization≈1.07 — close but not decisively governing. Decision: since concept stage uses one constant wall thickness while a real design would taper it (thicker at mudline, thinner toward the interface), refining a buckling check now would be precision spent on a thickness distribution later stages will change anyway.

## Open questions
- **Buckling deliberately not implemented** (2026-07-17) — local shell buckling utilization was ≈1.07 at a realistic 10m section length in a hand check, close enough to governing that it's worth revisiting once the model supports a thickness taper and real section/can lengths are known; global buckling is unlikely to ever matter. Not tracked as a bug — a scoped-out concept-stage decision.
- FATIGUE_LOAD_FACTOR=0.176 is still ad-hoc (chosen via a sensitivity sweep targeting a true-minimum thickness of ~60mm at the 15MW reference case, excluding NFA, not derived from a DLC spectrum) — recalibrate later against real fatigue load data.
- **NFA is not yet verified** (user-flagged, 2026-07-16) — separate from the f0-formula check already done (§7/§10 below). With the thickness-shrink fix, several converged designs now sit within a hair of their NFA limit (margin as low as 0.002), so this has become higher-priority to resolve.
- The thickness-shrink step only shrinks wall thickness, not diameter — a design could still be over-conservative on diameter for a given case; left for later since diameter also feeds NFA and the extreme-load calc.
- Sand nh-vs-friction-angle table and clay k=0.25×su correlation are rough approximations, not validated against site-specific geotechnical data.
- NFA's f0 prediction is checked against 4 real turbine sizes (5/15/20/22MW, see METHODOLOGY_REPORT.md §10) but still omits the K_LM cross-coupling term and uses a single-point-calibrated tower regression for the tower segment above the transition piece — and, separately, the check as a whole is not yet considered verified (see above).
